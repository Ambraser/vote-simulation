# V1.0.0

First release of `vote_simulation`, a simulation framework built on top of [SVVAMP](https://github.com/francois-durand/svvamp) for large-scale experiments on social choice rules.

---

## Simulation engine

The core engine (`simulation.py`) orchestrates the full experiment pipeline:

1. Load a TOML configuration file.
2. For each combination of generative model × number of voters × number of candidates × iteration: load a cached profile from disk (Parquet) or generate and persist a new one.
3. Apply every configured voting rule to each profile and collect results.
4. Persist simulation results under `data/sim_result/<MODEL>_v<NV>_c<NC>/iter_NNNN.parquet`.

Generated profiles are cached under `data/gen/<MODEL>_v<NV>_c<NC>/iter_NNNN.parquet` and reused across runs.  A `reload` flag forces regeneration.

### Configuration (`SimulationConfig`)

Simulation plans are defined in a TOML file (see template: `config/simulation.toml`)

## Voting rules registry

`models/rules/registry.py` maps short codes to SVVAMP rule factories.  The following rules are available out of the box:

**Score-based**: Plurality, Borda, Approval, K-Approval, Range Voting, Majority Judgment, Veto, STAR

**Elimination-based**: IRV, Two-Round, Baldwin, Nanson, Coombs, Bucklin, Iterated Bucklin, Exhaustive Ballot, IRV Average, IRV Duels

**Condorcet-based**: Copeland, Maximin, Schulze, Kemeny, Ranked Pairs (Tideman), Slater, Split Cycle, Woodall, Young, Kim–Roush, Black, Dodgson, Condorcet-IRV variants (CondorcetAbsIRV, CondorcetVtbIRV, CondorcetSumDefeats), Smith-IRV, ICRV

Custom rules can be added by calling `register_rule(code, builder)` with a callable matching the `RuleBuilder` protocol.

### Co-winner resolution

When the primary winner is ambiguous (tie), the engine resolves co-winners according to the family of the voting rule (score based, elimination based ...). Rules are wrapped individually to ensure the correct behavior.

## Winner metrics

`models/rules/winner_metrics.py` computes a rich set of winner-quality metrics from the SVVAMP `Profile` and the co-winner index set.  All computations are fully vectorised with NumPy.

| Metric | Description |
|---|---|
| `social_acceptability` | Fraction of voters giving utility > 0.5 to at least one co-winner |
| `utility_mean` / `utility_median` / `utility_var` | Distribution of utilities across all (voter, co-winner) pairs |
| `rank_mean` / `rank_median` / `rank_var` | Distribution of 1-based ranks across all (voter, co-winner) pairs |
| `freq_first` | Fraction of voters who rank at least one co-winner first |
| `freq_last` | Fraction of voters who rank at least one co-winner last |
| `has_tie` | `True` when there are multiple co-winners |
| `n_cowinners` | Number of co-winners |

When multiple co-winners exist, per-voter series are stacked across all winners so metrics reflect the whole winning set.

## Result models

| Class | Description |
|---|---|
| `SimulationStepResult` | Result for a single (profile, rule) pair |
| `SimulationSeriesResult` | Aggregated results for one `(model, n_voters, n_candidates)` series across all iterations and rules |
| `SimulationTotalResult` | Top-level container aggregating multiple series across the full parameter space; supports filtering, scalar-metric pivot tables, and heatmap/comparison-grid plots |

### Distance metrics

Rule-outcome similarity is measured with **Binary** and **Jaccard** distances, both exposed through a base class that users can extend to define custom distance functions. Results are summarised in distance matrices.

## Streamlit UI

A graphical interface is available via:

```bash
vote-sim-ui
# or
streamlit run src/vote_simulation/ui/app.py
```

The UI is organised into four tabs:

| Tab | Description |
|---|---|
| **Configuration** | Load, edit, and save TOML configuration files; shows the currently active config path |
| **Data generation** | Configure and trigger profile generation for the selected models, voter counts, and candidate counts; supports per-model extra parameters |
| **Simulation** | Select rules by family, launch the full simulation run with live progress and log streaming, stop a running job |
| **Results** | Browse and visualise simulation outputs: metric heatmaps, comparison grids, distance matrices |

The UI runs simulation jobs in a background thread with a stop event and a live log queue so the interface stays responsive during long runs.
